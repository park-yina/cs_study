# 기존 사용 경험
- svn과 깃의 경우 각각의 장단점이 분명히 있음
- 임베디드 개발자분과 협업을 하며 git을 잘 쓰지 않는다는 말을 듣고 놀랐는데 이는 svn이 더 큰 용량을 지원해주기 때문

# svn의 장점
- 앞서 말한 바와 같이 대용량 파일의 업로드에도 자유로움
- 다른 개발자의 커밋과 쉽게 얽히지 않는 중앙집권체제적 특성을 가지고 있으므로 롤백이 깃에 비해 쉬울 수 있음(깃도 특정 버전 고르는 체리픽 존재는 함)

# 깃과 svn의 체감되는 특징(실질적인 사용후기)
- 버전에 따른 코드를 관리할 수 있음
    - 하지만 성격이 다름. svn의 경우 중앙에서 데이터를 가지고 있으므로 여러 개발자 동시에 작업을 할 때에 원본 데이터의 훼손을 깃에 비해서는 막을 수 있음(각각 개발자가 다른 파일을 자신의 컴퓨터에 내려받는 형식이기 때문)
    - 깃의 경우 반대로 main브런치를 통해 `중앙 관리 시스템`의 느낌을 낼 수는 있지만, 근본적으로는 여러 브런를 활용해 보다 유연성 높은 작업을 제공함
- 그렇다면 git은 한번 revert한 커밋에 대해 롤백이 불가능한가?
    - 그것은 아님. 체리픽 등을 통해 revert이전의 특정 버전으로 돌아갈 수도 있음. 하지만 svn에서는 서버에서 개인이 내려받은 파일을 관리하므로 누군가가 다시 서버에 원본 파일을 올리면 해결되는 문제가 깃에서는 보다 복잡한 해결방식을 요구할 수 있음

# SVN vs Git 사용 경험

## 기존 경험
- SVN과 Git 모두 장단점이 있음
- 임베디드 개발자와 협업 시 Git을 잘 쓰지 않는 이유: SVN이 대용량 파일을 더 잘 지원함

## SVN 장점
- 대용량 파일 업로드 가능
- 중앙집중형 구조로 다른 개발자 커밋과 충돌이 적어 롤백이 직관적

## Git 장점
- 로컬 저장소 기반으로 브랜치를 자유롭게 활용 가능
- 유연한 협업과 병합 지원

## 체감되는 특징 비교
| 항목 | SVN | Git |
|------|-----|-----|
| 구조 | 중앙집중형 | 분산형(로컬 저장소) |
| 협업 | 동시에 여러 개발자가 작업해도 원본 데이터 보호 용이 | 브랜치를 활용해 유연하게 작업 가능, 충돌 관리 필요 |
| 롤백 | 서버에서 내려받은 파일로 쉽게 복원 | revert, cherry-pick으로 특정 버전으로 복원 가능, 과정이 상대적으로 복잡 |
| 대용량 파일 | 강점 | 상대적으로 제한적 |
